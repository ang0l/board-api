## Введение

### О FastAPI

#### Введение в FastAPI

**Популярность**. FastAPI быстро стал популярным благодаря своим архитектурынм и производительным преимуществам. Он появлися в 2018 году и уже обошел такие фреймворки, как Flask и Django.

#### Историческая справка

**Django**: появился в 2005 году, подходит для полноценных тяжелых приложений, но избыточен и менее производителен для API.

**Flask**: появился в 2010 году, минималистичен, но требует дополнительных усилий для расширения и поддержки архитектуры.

#### Преимущества FastAPI

**Типизация как архитектура**: позволяет легко масштабировать и рефакторить приложение.
**Асинхронность**: встроена из коробки, упрощая создание асинхронных операций.
**Автоматическая документация**: генерируется автоматически, упрощая интеграцию с фронтенд-разработкой.
**Dependency Injection**: обеспечивает простоту в управлении зависимостями и интеграции сервисов.
**Производительность**: выше, чем у Django и Node.js, с меньшей задержкой и использованием памяти.

#### Реальные примеры использования

FastAPI используется крупными компаниями, такими как Google, Microsoft, Netflix, Яндекс и другими.

#### Требования к слушателям

## Настройка окружения

### UV

Преимущества UV и как с ним работать.

#### Сравнеение с Poetry и PIP

UV опережает Poetry по производительности. Совместим с PIP и предлагает управление пакетами и проектами.

#### Функционал UV

Поддерживает Project TOML. Автоматически управляет окружениями для Python-проектов. Установка и публикацйия пакетов. Поддержка воркспейсов и кэшей.

#### Установка UV

На macOS/Linux через curl, на Windows через простую команду. После установки доступен интерфейс помощи.

#### Использование UV

`uv init`: инициализирует новый проект с созданием файлов _main.py_ и _pyproject.toml_.
`uv add`: добавление пакетов зависиомостей с автоматической настройкой окружения.
`uv run`: запуск установленных пакетов.
Логирование зависимостей через logfile для фиксации версий.
`uv sync`: синйронизация зависимостей.

#### Преимущества

Автоматическое управление окуржениями. Упрощение работы с зависимостями. Унификация инструментов управления проектами.

**В итоге**, UV заменяет необходимость использования нескольк4их инструментов, таких как Poetry и другие, что упрощает управление проектами и ускоряет процесс разработки.

#### Установление основ проекта

Создается директория под названием `board-api`. Инициализируется проект с помощью команды `uv init`. Структура будет содержать файлы: _main.py_, _pyproject.toml_, и стандартный _README_.

#### Организация структуры кода

Создаем папку _app_ и перемещаем туда _main.py_ для лучшей организации кода проекта.

Эта структура проекта с помощью UV позволяет эффективно разрабатывать и развивать приложение на FastAPI, модернизирую подход благодаря использованию современных инструментов.

### Uvicorn

#### Основная команда

Попробуйте запустить файл _main.py_ с помощью Python:

```
py main.py
```

Однако простое выполнение файла не запустит сервер для обработки входящих HTTP-запросов. Испольование команды `uv run` (или любого другого способа просто выполнить скрипт) также не приведет к запуску веб-сервера, поскольку FastAPI не является сервером - это фреймворк.

#### FastAPI CLI

Для запуска FastAPI небоходимо использовать CLI-инструмент или ASGI-сервер (например, _uvicorn_). Под капотом используется ASGI-сервер, который:

- открывает сетевой порт;
- принимает соединения;
- передает запросы приложению;
- отправляет ответы клиенту.

#### Правильный запуск

Используйте _uvicorn_ в текущем окружении. Укажите режим разработки с флагом `--reload`. Определите точку входа приложения в формате:

```
module_name:app_instance
```

Пример:

```
uvicorn main:app --reload
```

Вы сталкнетесь с ошибкой, если в файле _main.py_ не определен объект приложения (`app = FastAPI()`); указано неверное имя переменой; неверно указан модуль при запуске.

#### Зачем запускать через ASGI-сервер

**Что такое Uvicorn**? Uvicorn - это ASGI-сервер (Asychronous Server Gateway Interface server).

Он:

- управляет event loop;
- принимает HTTP-соединения;
- передает запросы приложению;
- возвращает ответы клиенту.

#### Преимущества ASGI

В отличие от WSGI (который обычно использует поток или процесс на каждый запрос - в зависимости от реализации сервера), ASGI работает через event loop.

Это позволяет:

- эффективно обрабатывать большое количество одновременных соединений;
- поддерживать асинхронность;
- работать с HTTP, WebSocket и Server-Sent Events (SSE).

ASGI лучше подходит для современных API и realtime-приложений.

#### Работа ASGI-сервера

**Обработка запросов**

1. Клиент отправляет HTTP-запрос.
2. ASGI-сервер принимает соединение.
3. Формируется отбъект _scope_.
4. Сервер вызывает приложение FastAPI.
5. FastAPI формирует ответ.
6. Ответ отправляется клиенту

**Ключевые элементы ASGI**

**Scope** - словать с описанием запроса (тип, путь, заголовки и т.д.).
**Receive** - корутина для получения данных от клиента.
**Send** - корутина для отправки данных клиенту.

**Жизненных цикл**. Клиент отправляет запрос - создается _scope_. FastAPI обрабатывает запрос. Ответ передается через _send_. Соединение либо закрывается (обычный HTTP), либо остается открытым (например, WebSocket).

#### Эффективность и производительность

Вместо создания отдельного потока или процесса на каждый запрос (как часто реализуется в WSGI-подходах), один процесс с event loop может обслуживать множество соединений. Event loop управляет выполнением асинхронного кода для каждого запроса.

#### Выводы

- FastAPI - это фреймворк, а не сервер.
- Для работы API всегда требуется ASGI-сервер.
- Высокая производительность достигается за счет неблокирующей модели обработки запросов.
